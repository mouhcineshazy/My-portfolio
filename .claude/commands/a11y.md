# /a11y — Accessibility Audit

You are a Senior Frontend Architect. Run a WCAG 2.1 AA accessibility audit of this portfolio. A portfolio is a professional showcase — inaccessibility here reflects on the developer's craft. Every finding matters.

## Audit scope

If a component path is given as an argument, audit only that file. Otherwise audit the full page.

Files to read for a full audit:
- `app/layout.tsx`
- `app/page.tsx`
- All files in `components/`
- `app/globals.css` (for focus ring styles)

## Checks

### Semantic structure
- `<html lang="...">` set to the active locale (currently hardcoded `"en"` — should react to the locale context)
- `<main>`, `<header>`, `<footer>`, `<nav>` present and used correctly
- Heading hierarchy: one `<h1>` per page, `<h2>` for sections, `<h3>` for sub-items — no skipped levels
- `<ul>` / `<li>` for the nav links and skills list — already done; verify
- `<section>` elements have `aria-labelledby` pointing to their heading `id`, or have an `aria-label`

### Keyboard navigation
- Tab order follows the visual reading order
- All interactive elements reachable by keyboard: links, buttons, form inputs
- Focus ring visible on every focused element (Tailwind `focus:ring-*` or `outline-none focus:outline-*`)
- No focus trap outside a modal/dialog
- The theme toggle (bottom-right button) is reachable and operable by keyboard
- The language toggle is reachable and operable by keyboard

### Images and icons
- Every `<Image>` has a non-empty, descriptive `alt` (not "image", not the filename)
- Decorative images have `alt=""`
- Icon-only buttons have `aria-label` — check:
  - LinkedIn button in `intro.tsx`
  - GitHub button in `intro.tsx`
  - ThemeSwitch button in `theme-switch.tsx`
  - LanguageSwitch button in `language-switch.tsx`

### Forms
- Every `<input>` and `<textarea>` has an associated `<label>` (via `htmlFor` / `id` pairing OR `aria-label`)
- Required fields are marked with `required` attribute AND visually indicated
- Error messages are announced to screen readers (use `role="alert"` or `aria-live="polite"`)
- The submit button has a visible label ("Submit" is fine, but "Send message" is better)

### Colour contrast (manual check — describe findings)
- Body text on light background: Tailwind `text-gray-950` on `bg-gray-50` — check ratio
- Body text on dark background: `text-gray-50/90` on `bg-gray-900` — check ratio
- Muted text (`text-gray-500`, `text-gray-700`) — these are high-risk for contrast failures
- Tag pills: white text on `bg-black/70` — check ratio
- Skill badges: `text-gray-800` on `bg-white` — check ratio

### Motion / animation
- Check for `prefers-reduced-motion` media query respect:
  - Framer Motion respects this via the `useReducedMotion` hook
  - None of the animation variants currently check `useReducedMotion` — this is a finding
  - Fix: import `useReducedMotion` from `framer-motion`, disable or minimise animations when `true`

### ARIA
- No ARIA roles that override semantic HTML unnecessarily
- `aria-expanded`, `aria-controls` on any disclosure widgets
- `aria-current="page"` on the active nav link (currently handled visually — needs ARIA too)

### Language
- `<html lang>` should update when the user switches to French (`lang="fr"`)

## Output format

```
[WCAG CRITERION] Component:line — Finding
  Impact: who is affected (screen reader users / keyboard users / low-vision users / motor-impaired users)
  Fix: specific change with code snippet
```

Criteria format: `1.1.1 Non-text Content` / `2.1.1 Keyboard` / `4.1.2 Name, Role, Value` etc.

End with:
- Count of WCAG A failures (must fix)
- Count of WCAG AA failures (should fix)
- Top 3 highest-impact fixes
